Non-blocking observations (pre-existing, out of spec scope):
- `lib/services/schedule.service.ts:23-28` — `classId` and `teacherId` both spread `{ course: ... }` into the same `where` key; if a caller passes both, the second spread silently overwrites the first (JS object spread semantics). The current callers in `app/api/lms/schedule-slots/route.ts:47-53` use mutually-exclusive `if/else if`, so the collision is latent, not triggered. Builder did not introduce this — the pattern was already present in HEAD (confirmed via `git show HEAD:lib/services/schedule.service.ts`). Not a PR-fix-2 regression. Worth a separate micro-PR later (or a single-line combine using explicit `AND` if the two filters ever need to co-exist).
- `app/api/lms/announcements/route.ts:50` has no `teacherId` branch for teacher users visiting `/announcements` without a courseId — they would see all published announcements across the system. Out of PR scope but flagging as potential B5 follow-up if teacher-side view ever materializes.

Static SQL audit confirms correctness against `CourseClass` DDL and schema.prisma; `INSERT ... SELECT ... ON CONFLICT DO NOTHING` will be safe to apply.
